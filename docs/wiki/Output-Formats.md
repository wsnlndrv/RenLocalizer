# 📄 Understanding Output Formats & Architecture

RenLocalizer uses a **Hybrid Output Architecture** (introduced in v2.7+) to ensure maximum compatibility with the Ren'Py engine while providing modern features like runtime translation and atomic segments.

---

## 🏗️ The Hybrid Trinity (v4.1.0 Standard)

When you run a translation, RenLocalizer generates three distinct components in your `game/tl/[language]/` folder:

### 1. Updated `.rpy` Files (The Body)
Traditional Ren'Py translation blocks.
*   **Purpose:** Permanent, file-based storage of your translations.
*   **Mechanism:** Uses the standard `old/new` or `translate [lang] [id]` syntax.
*   **Note:** These are what Ren'Py officially loads on startup.

### 2. `strings.json` (The Brain)
A high-performance Translation Database.
*   **Purpose:** Stores every single translated string as a Key-Value pair.
*   **Mechanism:** Used by the **Runtime Hook** for instant lookups.
*   **Benefit:** Enables translation of dynamic strings, hard-coded Python variables, and technical UI elements that `.rpy` files sometimes miss.

### 3. `_rl_hook.rpy` (The Soul)
The "Oyun-İçi Hook" (In-game bridge).
*   **Purpose:** Injects RenLocalizer logic directly into the Ren'Py engine.
*   **Mechanism:** Overrides the `say_menu_text_filter` and `replace_text` globally.
*   **Benefit:** Guarantees that even strings generated by complex code functions are translated without crashing the game.

---

## 🔹 Atomic Segments (Angle-Pipe System)

RenLocalizer v2.7.2+ introduces **Delimiter-Aware Translation** for strings like `<Option A|Option B|Option C>`.

*   **Logic:** The tool splits these into individual segments (`Option A`, `Option B`), translates them separately, and then rejoins them.
*   **Output:** Each segment is saved as an **Atomic Entry** in `strings.json`.
*   **Benefit:** Perfect compatibility with Ren'Py `vary()` calls and complex menu logic.

---

## 🔹 Transfer & Interchange Formats

These are **not** part of the game files but are used in the **Tools** tab for external workflows:

| Format | Extension | Use Case |
|---|---|---|
| **Structured JSON** | `.json` | Best for LLM APIs (GPT-4, Claude) and developers. |
| **Simple TXT** | `.txt` | Best for "Copy-Paste" into AI web chats (Free versions). |
| **Excel/Spreadsheet** | `.xlsx` | Best for professional human translators. |
| **TM Library** | `.tm` | A binary format for the RenLocalizer Translation Memory system. |

---

## 📂 Final Directory Layout
All game-ready output is placed here:
`game/tl/[target_language]/`
- `script.rpy` (File-based blocks)
- `strings.json` (The DB)
- `_rl_hook.rpy` (The System Hook)

---

> [!TIP]
> **Pro Tip:** Never delete `strings.json` or `_rl_hook.rpy` manually. If you do, the "Hard-to-reach" technical strings in your game will return to English.
